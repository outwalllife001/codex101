高级技巧

提示词工程

四大要素框架

有效的提示应包含四个要素：

要素 说明 示例
目标 你想改变或构建什么？ "添加用户认证功能"
上下文 哪些文件、错误相关？ "@auth.ts @user.ts"
约束 应遵循的标准？ "使用 JWT，不要 session"
完成标准 什么条件下算完成？ "测试通过，能登录注册"

示例提示

目标：重构认证模块，支持 JWT token 刷新
上下文：@src/auth/ 目录下的所有文件
约束：
- 使用纯 JWT，不要 session
- 保持向后兼容
- 添加 rate limiting
完成标准：
- 所有现有测试通过
- 新增 token 刷新测试
- 手动测试登录流程正常

Git 工作流集成

使用 Git Checkpoints

Codex 可能大幅修改代码库，建议：

  1. 开始任务前创建 checkpoint：git add -A && git commit -m "checkpoint: before codex auth refactor"
  2. 任务完成后 review diff
  3. 必要时可以回滚：
    git reset --hard HEAD~1

使用 Git Worktrees

当需要在同一代码库上运行多个 Codex 线程时：

# 创建 worktree
git worktree add ../my-project-feature-a feature-a
git worktree add ../my-project-feature-b feature-b

# 在不同目录启动 Codex
cd ../my-project-feature-a && codex
cd ../my-project-feature-b && codex

脚本化 Codex

非交互式执行

# 执行单个任务
codex exec "修复 CI 失败"

# 安静模式
codex exec --quiet "更新依赖"

# 结合 shell 脚本
for file in src/**/*.test.ts; do
  codex exec "检查 $file 的测试覆盖率"
done

# .github/workflows/codex-review.yml

Advanced Tips

Prompt Engineering

Four Elements Framework

An effective prompt should contain four elements:

Element Description Example
Goal What do you want to change or build? "Add user authentication feature"
Context Which files or errors are relevant? "@auth.ts @user.ts"
Constraints What standards should be followed? "Use JWT, no sessions"
Completion Criteria Under what conditions is it considered done? "Tests pass, login and registration work"

Example Prompt

Goal: Refactor the auth module to support JWT token refresh
Context: All files under the @src/auth/ directory
Constraints:
- Use pure JWT, no sessions
- Maintain backward compatibility
- Add rate limiting
Completion Criteria:
- All existing tests pass
- New token refresh tests added
- Manual testing confirms login flow works

Planning Mode

When to Use /plan

  • Changes involving multiple files
  • Need to modify architecture or interfaces
  • Unsure about the best implementation approach
  • Task may take a long time

パフォーマンス最適化
